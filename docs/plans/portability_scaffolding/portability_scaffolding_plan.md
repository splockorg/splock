## Problem statement

splock's Phase-0 portability scaffolding introduces host-neutral seams so the tool can later run under non-Claude hosts and transports without a rewrite, while shipping zero observable Claude Code behavior change today. The plan reconciles: (1) a frozen host-neutral hook event vocabulary (`bin/_host/events.py`) that every downstream port shares; (2) a deny-parity gate that must hold before the Hook Shim `hook-entry` trampoline cutover; (3) the two distinct meanings of 'host' — the RDS 'host session id' flag in the Transcript Provider (SC3) versus the Claude Code session-identity env vars scrubbed at the model-transport spawn boundary (SC4) — which do not collide; (4) a fixed-verifier invariant and roster `schema_version` 4 for ModelCatalog role→pin resolution (SC5); (5) the CLI-dialect `sanitize_schema` transform is empirically proven (a canned gpt-5.6 emission validated against the ORIGINAL strict schema, pinned by `test_sanitize_schema.py` against golden schemas including `plan_v1.schema.json`); and (6) the nested-session deadlock is empirically confirmed (2026-07-22): SDK-spawning splock commands hang when invoked inside a Claude Code session because the nested `claude` inherits eight session-identity env vars — so ClaudeTransport/`_sdk_bridge` MUST scrub `CLAUDECODE, CLAUDE_CODE_SESSION_ID, CLAUDE_CODE_CHILD_SESSION, CLAUDE_CODE_BRIDGE_SESSION_ID, CLAUDE_CODE_ENTRYPOINT, CLAUDE_CODE_EXECPATH, CLAUDE_PID, CLAUDE_EFFORT` at the spawn boundary (a build-now SC4 item, not a reservation; no-op when unset, preserving `CLAUDE_PLUGIN_*`); and the CLI-dialect `sanitize_schema` helper ships now as a tested pure function (the earlier operator-confirmation hedge is resolved to build-now), while CodexTransport stays deferred.

## Conceptual architecture

Three host-portability ports (HostHookShim, TranscriptProvider, ModelTransport) plus a shared stdlib-only boundary vocabulary (bin/_host/events.py), a keyed transport registry and a StaticRouter-only routing seam (bin/_host/registry.py + routing.py), a ClaudeModelCatalog with role→pin data relocated into agents/_roster.json (schema_version 4), additive routing metadata in skill frontmatter and the intent registry, a documented reservations set, and an incrementally-built tests/host_conformance/ suite. Claude Code remains the sole live host; ClaudeTransport, ClaudeHookShim, and ClaudeTranscriptProvider WRAP (do not replace) the proven _sdk_bridge/sdk_spawners/hook plumbing; StaticRouter selects family only and can never override the fixed verifier pin. Every change is gated by zero-behavior-change-on-Claude-Code plus CI-green at each landing, with deny-parity, sanitize round-trips, and the catalog contract as the affirmative regression proofs. Suggested implplan phase grouping under the standing CI-green invariant: Phase 0a = SC1+SC4+SC5 (transport substrate); Phase 0b = SC2+SC3 (hook/transcript extraction behind the deny-parity gate); Phase 0c = SC6+SC7+SC8 (routing seam + metadata + registry fields); Phase 0d = SC9+SC10 close-out.
